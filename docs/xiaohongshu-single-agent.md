# 小红书单 Agent 案例说明

## 为什么这次不用多 Agent

参考文章里的思路，完整业务链路是成立的：素材收集、筛选分析、内容创作、图文产出，确实很适合做成多 Agent 系统。

但这次分享目标不一样：

- **优先要稳定演示**，而不是展示最复杂架构
- **优先要让观众听懂**，而不是一上来讲角色拆分
- **优先要现场跑通闭环**，而不是依赖外部平台状态

所以这里特意把方案收敛成：

> 一个 Agent，串行完成 collect → analyze → draft → package

这样做的好处：

1. 演示路径短，出错点少
2. 输入、处理中间态、输出都更容易解释
3. 以后仍然可以平滑升级成多 Agent

## 当前 demo 包含什么

页面入口：`/ops/xiaohongshu`

运行一次 campaign 后，会得到：

- `campaign`：一次内容生产任务
- `topics`：候选与入选话题
- `runs`：四个阶段的执行记录
- `notes`：生成的笔记草稿
- `packages`：最终物料包元数据

## 四个阶段如何映射业务

### 1. collect
模拟收集热点或竞品样本，产出候选话题池。

当前为了保证 demo 稳定，使用 mock + 规则生成，不依赖真实第三方 API。

后续可替换为：

- 自定义抓取 skill
- 搜索 API
- 飞书多维表格写入
- 小红书热点/搜索结果采集脚本

### 2. analyze
根据分享场景和话题得分，选出更适合生成的主题。

当前策略是规则打分与排序。

后续可替换为：

- 真实 LLM 分析
- 基于历史爆文数据的排序模型
- 飞书数据回读后二次筛选

### 3. draft
生成标题、hook、正文、封面文案、分页卡片结构。

当前实现是模板化生成，强调“能演示完整闭环”。

后续可替换为：

- Auto-Redbook-Skills
- 真实大模型创作
- 合规提示词模板

### 4. package
把产物整理成一份可交付的物料包：输出目录、文件列表、发布检查项。

当前输出的是本地元数据。

后续可替换为：

- Markdown 渲染图片
- 批量导出 PNG/JPG
- 发布脚本
- 自动同步到飞书文档或云盘

## 和文章里的多 Agent 方案差异

| 维度 | 文章方案 | 当前 demo |
|---|---|---|
| 架构 | Director + Collector + Creator | 单 Agent 串行流水线 |
| 依赖 | 外部 skill / 飞书 / 采集脚本 / 渲染 / 发布 | 本地 mock + SQLite + Next.js |
| 适用场景 | 真实生产系统 | 分享演示 / MVP 验证 |
| 复杂度 | 高 | 低 |
| 稳定性 | 取决于外部依赖 | 高，适合现场跑 |

## 后续怎么升级成真实系统

建议按下面顺序替换：

1. **先替换 collect**
   - 接入真实搜索 / 热榜 / 竞品抓取脚本
2. **再替换 draft**
   - 接入真实 LLM 和渲染 skill
3. **最后替换 package**
   - 输出真实图片、发布素材和飞书链接
4. **确认稳定后再拆 Agent**
   - 当 collect 和 draft 已经有独立资源瓶颈时，再考虑拆成多个 agent

## 演示建议

现场可以直接这样讲：

1. 输入一个关键词，比如 `openclaw`
2. 选择模式：热点追踪 / 竞品分析
3. 选择生成篇数
4. 点击运行
5. 展示四个阶段的结果
6. 展示最终笔记和物料包

一句话总结：

> 这不是“最复杂”的架构，但它是今晚最适合现场讲透、讲稳、讲明白的架构。
